# 深夜バッチ運用プロトコル(3時間×2ターン制)
### 〜無人環境でJulesを最大20タスク/夜スケールさせ、翌朝トークン最小で検収する〜

> 大前提となる安全設計: **深夜に無人でマージしてよいのは「使い捨ての統合ブランチ」まで。mainへのマージは翌朝の人間に残す。** これにより第2部のフェイルセーフ(不可逆操作の人間承認)を破らずに、無人でのターン間依存解決を実現します。

---

## 0. 全体像(タイムライン)

```
[23:00 就寝前 15〜30分] ★人間+建築家が唯一介在する仕込みフェーズ
  - タスクマニフェスト(tasks.yml)の確定
  - integration/nightly-YYYYMMDD ブランチ作成
  - 第1ターンのタスク投入 + Jules計画(Plan)の承認 ←ここだけ人間が計画を見る

[23:30〜02:30 第1ターン] 土台タスク 最大10並列(無人)
  - PRは integration ブランチ宛て
  - CI合格 + 機械検査合格のPRのみ、Actionsがintegrationへ自動マージ

[02:30 ターン切替] GitHub Actions(スケジュール実行、無人)
  - 依存元がマージ済みの第2ターンタスクのみ、Jules APIで投入(fail-closed)
  - 依存元が失敗したタスクはスキップし、ログに記録

[02:30〜05:30 第2ターン] 応用タスク 最大10並列(無人)
  - integration の最新状態を起点に実装。PRは integration 宛て

[翌朝] 一括検収
  - 機械プリフィルタ(トークン0) → Fable 5一括スクリーニング(1回)
  → 深掘りが必要な少数のみ個別レビュー → 人間が integration→main をマージ
```

## 1. 2ターン制のタスク分割・配置アルゴリズム

### 仕分けの4基準(この順に適用)

**基準1: 深夜投入の適格性(そもそも夜に走らせてよいか)**
無人運転では計画承認も途中修正もできない(放置すると計画は自動承認される)ため、夜間タスクは以下に限定します:
- ✅ 適格: テスト追加、ドキュメント、独立した新規機能、明確な仕様のバグ修正、型定義・ユーティリティ追加
- ❌ 不適格(日中の対話運用に回す): DBマイグレーション、本番設定・CI設定・認証周りの変更、破壊的リファクタ、仕様に曖昧さが残るタスク
- 判定質問: **「最悪の実装が出てきても、翌朝ブランチごと捨てれば無傷か?」** — Noなら夜に流さない。

**基準2: ファイル所有権による衝突判定(同一ターン内の並列可否)**
- 各タスクに「書き込み対象パス(Issueのスコープ欄)」を宣言させ、**同一ターン内の10タスクは書き込みパスが互いに素(disjoint)であること**を機械的に検証する。1ファイル=1オーナーの原則。
- 共有ファイル(index.ts、ルーティング表、依存定義ファイル等)に触るタスクは各ターン1件まで。どうしても複数必要なら、共有部分の変更だけを第1ターンの1タスクに集約する。
- コンフリクトは「起きたら解決」ではなく「配置で起こさない」。無人環境でのコンフリクト解決は品質の墓場です。

**基準3: 依存方向(どちらのターンに置くか)**
- **第1ターン=生産者(Producer)**: 他タスクが利用する土台 — 共通インターフェース・型定義、共有ユーティリティ、APIのスキーマとスタブ、テスト基盤・fixture
- **第2ターン=消費者(Consumer)**: 第1ターンの成果を import して使う実装 — 各エンドポイントの実装、UIコンポーネント、土台を使うテスト群
- **依存の深さは1ホップまで。** T1内でタスクAがタスクBに依存する配置は禁止(同一ターン内は完全独立)。2ホップ必要な連鎖(A→B→C)は、Cを翌日に回す。1晩で解決する依存を欲張らないことが安定の鍵。

**基準4: 不確実性の前倒し**
- 成否が読みにくい探索的タスクは第1ターンに置く。失敗しても第2ターンの枠を無駄にせず、翌朝の判断材料が最大化される。逆に確実に通る定型タスク(docs等)は第2ターンの埋め草に最適。

### 配置の実務(キャパシティ設計)

- 理論上限は 10並列×2ターン=20タスク/夜(日次100枠の20%)。ただし**立ち上げ期は 1ターン5件程度から始め、翌朝の一発合格率が7割を超えてから増枠**してください。低品質PRを20件量産すると、翌朝のレビューコストが夜間の節約を食い潰します。
- 3時間で終わらない規模のタスクは分割対象。目安として「計画ステップが多すぎる/変更ファイルが多数に及ぶ」ものは夜間不適格です。

### タスクマニフェスト(tasks.yml) — 配置判断を機械可読にする

就寝前に建築家(私)が生成し、リポジトリに置く「夜の設計図」です。ターン切替Actionと翌朝レビューの両方がこれを参照します。

```yaml
night: 2026-07-14
integration_branch: integration/nightly-20260714
turn1:
  - id: T1-01
    issue: 101            # GitHub Issue番号
    paths: ["src/lib/validation/**"]   # 書き込み許可パス(互いに素であること)
    risk: low
  - id: T1-02
    issue: 102
    paths: ["src/types/order.ts", "src/api/schema/**"]
    risk: low
turn2:
  - id: T2-01
    issue: 111
    depends_on: [T1-01, T1-02]   # 依存元が全てintegrationにマージ済みの時のみ投入
    paths: ["src/api/orders/**"]
  - id: T2-02
    issue: 112
    depends_on: []               # 独立タスクはT2の空き枠に充填
    paths: ["docs/**"]
```

## 2. 無人での「ターンの切り替え」メカニズム

### 中核となるブランチ戦略: 「使い捨て統合ブランチ」方式

```
main(保護ブランチ、夜間は不可侵)
 └─ integration/nightly-0714(就寝前にmainから作成。夜の作業場)
     ├─ T1タスクのPR ──CI+機械検査合格──▶ integrationへ自動マージ
     └─ T2タスクのPR(integrationの最新=T1成果込みを起点に実装)
翌朝: 検収後、人間が integration → main のマージを実行
問題があった夜: integrationブランチごと削除 = 完全ロールバック
```

この方式の要点は3つです。
1. **無人自動マージの対象を統合ブランチに限定**することで、「マージは人間」の原則をmainについて維持したまま、T2がT1の成果を使えるようにする。統合ブランチへのマージは翌朝捨てられるため、実質的に可逆操作 [WRITE-R] に格下げされる。
2. mainにはブランチ保護(レビュー必須・管理者含む強制)をかけ、Actionsのトークン権限からもmainへのpushを外す。**「深夜に事故ってもmainは無傷」を権限側で物理的に担保する**(プロンプトは第2の防衛線、の原則)。
3. 日付入りブランチ名により、夜ごとの成果が独立し、ログ・監査・ロールバックが単純化する。

### GitHub Actionsの構成(3本のワークフロー)

**① auto-merge-to-integration.yml(T1 PRの無人マージ)**
- トリガー: `integration/nightly-*` 宛てPRのCI完了
- ゲート条件(全て満たす場合のみ自動マージ):
  - CI(テスト・Lint・ビルド)が全合格
  - 変更ファイルが tasks.yml 宣言の paths 内に収まっている(スコープ逸脱の機械検査)
  - 既存テストファイルの削除・変更が含まれない(含む場合は自動マージせず「要人間」ラベル)
  - PR作成者がJulesである
- 不合格PRはマージせず放置(翌朝レビューへ)。**ゲートは減点方式ではなく全会一致方式**にする。

**② turn-switch.yml(02:30のターン切替)**
- トリガー: schedule(cron)
- 処理:
  1. tasks.yml を読み、turn1 各タスクのintegrationへのマージ状況を確認
  2. turn2 の各タスクについて `depends_on` が**全てマージ済みの場合のみ**、Jules APIでセッションを作成して投入(起点ブランチ=integration)。APIには計画承認のエンドポイントもあるため、投入と承認方針をスクリプトで制御できる
  3. 依存が欠けたタスクはスキップし、skipped_tasks.json に理由を記録(fail-closed: 疑わしきは走らせない)
  4. 投入結果サマリをSlack等に通知(人間は寝ていて見ないが、翌朝の一次情報になる)

**③ morning-report.yml(05:45の朝会資料生成)**
- トリガー: schedule
- 処理: 全PRの「PR番号 / 対応タスクID / CI結果 / 変更ファイル数と行数 / スコープ逸脱有無 / テスト改変有無」を機械集計し、morning_report.md を生成。**これがセクション3のプリフィルタ入力になり、Fable 5のトークンを大幅節約する。**

### 代替案との比較(なぜこの方式か)

| 方式 | 利点 | 不採用/限定採用の理由 |
|------|------|------|
| **A. 統合ブランチ方式(推奨)** | mainが無傷。依存解決が可能。ロールバックが1操作 | — |
| B. スタックドブランチ(T2をT1のブランチ直上に積む) | マージ自体が不要 | T2が複数のT1成果に依存する場合に破綻。リベース地獄 |
| C. 完全独立並列(依存ゼロのタスクのみ流す) | 切替機構が一切不要で最も安全 | 依存解決という要件を満たさない。ただし**依存のない夜はこのCに退化させるのが正解**(ターン制は依存がある夜だけの装備) |
| D. main直マージの完全無人化 | 朝が楽 | 不可逆操作の無人化であり不採用。テスト改変等のすり抜けが本番に直行する |

## 3. 翌朝の「一括レビュー」プロトコル(トークン節約型)

### 節約の設計思想: 「LLMに読ませる前に、機械とルールで件数と分量を減らす」

**第0段: 機械プリフィルタ(トークン消費ゼロ)**
morning-report.md の機械集計だけで、以下は私に読ませる前に仕分けが確定します:
- CI不合格 → 自動で【要修正】(私のレビュー不要。CIログの要約だけ後で個別依頼)
- スコープ逸脱 or 既存テスト改変あり → 自動で【要人間確認】(重大注意フラグ付き)
- 残った「CI合格・スコープ内・テスト非改変」のPRだけが一括スクリーニングの対象

**第1段: 一括スクリーニング(私への依頼は原則1回)**
- 共通文脈(AGENTS.md・雛形・評価基準)は冒頭に1度だけ貼る。**AGENTS.mdは全文でなく、レビューに効く条項の抜粋(規約・禁止事項)で足ります。**
- Diffの渡し方は二層にする: **400行以下のPRはdiff全文、超えるPRは `git diff --stat` +主要ファイルのdiffのみ**を貼り、私の側から「追加でこのファイルを見たい」と指名させる(全文を先回りで貼るのが最大のトークン浪費源です)
- 出力は表+不合格理由のみに制限し、「問題のないコードの解説」を禁止する

**第2段: 深掘り個別レビュー(少数のみ)**
第1段で【保留】になったPRだけ、前章の重厚な個別レビュープロンプトにかける。経験則として20件中2〜4件に収まれば健全です。

### コピペ用・一括スクリーニングプロンプト

```markdown
あなたはこのプロジェクトの設計者として、昨夜の無人バッチでJulesが提出した
複数PRの一次スクリーニングを行ってください。目的は「精密な個別レビュー」では
なく「仕分け」です。出力は最小限に絞ってください。

## 共通文脈(全PRに適用)
### プロジェクト規約(AGENTS.md抜粋)
{{規約・禁止事項の抜粋}}
### 昨夜のタスクマニフェスト
{{tasks.yml を貼付}}
### 機械検査レポート
{{morning_report.md を貼付。CI不合格・スコープ逸脱のPRは既に除外済み}}

## 各PRのパケット(以下をPR件数分繰り返し)
=== PR #{{番号}} (タスク{{ID}}) ===
- Issueの完了条件: {{チェックリスト部分のみ貼付。Issue全文は不要}}
- Jules記載のPR説明文: {{貼付}}
- Diff: {{400行以下→全文 / 超過→ diff --stat と主要ファイルのみ}}

## 仕分け基準(3値)
- 【合格候補】完了条件を全て満たし、範囲逸脱・テスト形骸化・明白なバグ・
  セキュリティ懸念がない
- 【要修正】不備が明確で、Julesへの修正指示が1〜3行で書けるもの
- 【保留=個別レビュー行き】以下のいずれか:
  判断に迷う / 変更が広範 / PR説明とDiffが食い違う / 設計判断が必要 /
  diffが部分提供で確信が持てない

## 検査の力点(一次スクリーニングで見るべきもの)
1. 完了条件との照合(各PRで最優先)
2. テストの実質性: 追加テストが完了条件を本当に検証しているか
3. PR説明文とDiffの一致(誠実性チェック)
4. 明白なバグ・エラー処理欠落・入力検証漏れのみ(細かな品質指摘は書かない)

## 制約(トークン節約のため厳守)
- 合格候補のPRには判定と一行根拠以外、何も書かない。コードの解説・賞賛を禁止
- 静的に検証できない事項は個別に列挙せず「動作確認要」とだけ記す
- diffが部分提供のPRで判断に必要なファイルがあれば、レビューせずに
  「保留: {{ファイルパス}}の提示が必要」と書いて次のPRへ進む
- 確信が持てないものを【合格候補】に入れない。迷ったら【保留】(fail-closed)

## 出力形式
1. 仕分け表: | PR | タスク | 判定 | 一行根拠 |
2. 【要修正】各件: Julesへ転送可能な修正指示(診断付き・3行以内)
3. 【保留】各件: 個別レビューで確認すべき論点(1〜2点)と追加で必要な材料
4. 横断所見(3行以内): 複数PRに共通する問題パターンがあれば
   (→ AGENTS.md/Issue雛形の改善材料としてナレッジログへ)
```

### 翌朝フロー全体(人間の動線)

1. morning_report.md を一読(2分) — CI不合格分は修正指示だけJulesに返す
2. 上記プロンプトで一括スクリーニング(私への依頼1回)
3. 【要修正】の指示文をJulesの各タスクへ転送(日中の再試行へ)
4. 【保留】のみ個別レビュー(前章の重厚プロンプト)
5. 【合格候補】を人間がサンプリング確認 → **integration→main のマージは人間が実行**
6. 「横断所見」をナレッジログに記録(第3部・週次高速ループの原料)

## 4. 正直な注意点(この運用固有のリスク)

1. **無人運転は計画承認ゲートを失う運用です。** 就寝前にT1の計画だけは人間が確認できますが、02:30投入のT2の計画は誰も見ずに自動承認へ進みます。だからこそ基準1(夜間適格性)を厳格に守り、T2にはリスクの低いタスクだけを置いてください。ここを緩めるなら日中運用に切り替えるべきです。
2. **自動マージゲートの「全会一致方式」を絶対に緩めないこと。** 統合ブランチとはいえ、汚染されたT1成果の上にT2の10タスクが積み上がると、翌朝20件全てが検収不能になります。ゲート1個の妥協が夜全体を無駄にします。
3. **スケールは検収能力に合わせる。** 20タスク/夜はJulesの上限であって、あなたの翌朝の上限ではありません。一発合格率・差し戻し率のログ(第3部のフライホイール)を見ながら、「翌朝1時間で捌ける量」を実測で決めてください。
4. Jules APIはアルファ版であり仕様変更がありえます。ターン切替スクリプトは「API呼び出しが失敗したらT2を投入せず通知だけ残す」fail-closed設計にしておくと、API変更時も事故ではなく単なる空振りで済みます。
