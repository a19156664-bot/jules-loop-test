# タスク指示書テンプレート（Task Instruction Template）

指揮官（Antigravity）が実行部隊（Jules）にタスクを発行する際は、本テンプレートに従って指示書を作成します。

---

```markdown
## 目的 (Why)
- 【この変更がなぜ必要か、真の目的を1〜3文で明確に記述】

## 変更内容 (What)
- 【具体的に何をどう変更・実装するか】

## 完了条件 (Definition of Done) — すべて検証可能な形で記述
- [ ] 【条件1: 例 - POST /api/v1/orders で不正パラメータ時に400エラーとエラーコードE-101を返すこと】
- [ ] 【条件2: 例 - 上記の正常系・異常系ユニットテストが追加され、npm test がパスすること】
- [ ] 【条件3: 例 - 既存のAPIレスポンスの型互換性が維持されていること】

## スコープ（変更許可範囲）
- **変更許可パス**:
  - `src/controllers/orderController.ts`
  - `tests/unit/orderController.test.ts`
- **変更禁止パス**:
  - `src/config/**`
  - `.github/workflows/**`
- **今回やらないこと**:
  - DBスキーマの変更や別リファクタリング

## 技術的制約・参照パターン
- 使用すべき既存関数・共通モジュール: `src/utils/validator.ts`
- 参照すべき類似実装: `src/controllers/userController.ts`

## 詰まった場合の指示（撤退条件）
- 完了条件を満たせない技術的障害が発生した場合は、独断で迂回策を作らず、作成途中の状態をログに記録して報告・停止してください。
- 指示内容とコードの実態に矛盾を見つけた場合も、修正を止めて報告を優先してください。

## PR説明文に含めるべき必須項目
1. **完了条件との照合結果**（各条件を満たした具体的コード・テストの提示）
2. **変更ファイル一覧と変更理由**
3. **テスト実行結果のログ抜粋**
4. **未確定事項・妥協点**（無ければ「なし」と明記）
```
