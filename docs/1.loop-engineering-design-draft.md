# ループエンジニアリング設計図（草案 v0.1）
作成: 2026-07-17 / 建築家: Fable 5
状態: **草案** — 「⬜未確定」箇所は3夜目結果などで埋める

---

## 1. コンセプト

**設計→実装→レビュー→再設計のループを、すべてLLMで閉じる。**

```
Claude Code（司令塔）: 判断・レビュー・タスク設計。コードは一切書かない
Jules（大工）        : すべてのコード生成
GitHub Actions      : 機械的な配管（CI / auto-merge / turn-switch / report）
人間（オーナー）     : 方針設定と週次確認のみ
```

現行の夜間パイプラインは本構想の**プロトタイプ第1号**。
turn-switch 自動チェーンは司令塔導入後も「司令塔なしで2タスク連鎖する」安全弁として残す。

## 2. 全体構成図

```
[Windows タスクスケジューラ]
  30分ごと（⬜間隔は運用で調整）
      │
      ▼
[commander.ps1] ──起動──▶ [Claude Code ヘッドレス（claude -p）]
      │                        │ 読む: 憲法v2 / state.yml / ロードマップ / gh 相当の情報
      │                        ▼
      │                  ┌─ 状態判断 ─────────────────────────┐
      │                  │ A. Jules作業中     → 何もせず終了     │
      │                  │ B. PR完成・CI通過  → コードレビュー    │
      │                  │ C. レビュー合格    → 次タスク投入      │
      │                  │ D. レビュー不合格  → 修正指示 or 分割  │
      │                  │ E. タスク枯渇      → ロードマップから設計│
      │                  │ F. 異常検知        → ループ停止+通知   │
      │                  └────────────────────────────────────┘
      │                        │ 書く: state.yml / tasks.yml / レビューコメント
      ▼                        ▼
[ログ保存]              [git push → GitHub]
                               │
                               ▼
                    [Jules が次タスクを実装]
                               │
                    [PR → CI → auto-merge → (turn-switch)]
                               │
                    （30分後、司令塔が再び起動）
```

## 3. 状態管理（state.yml）

司令塔はステートレス起動のため、「前回の判断」をリポジトリ内ファイルに永続化する。

パス案: `.nightly/state.yml`（protected_paths 配下 = Jules は触れない）

```yaml
loop_status: running          # running / paused / stopped
last_commander_run: "2026-07-17T21:30:00+09:00"
current_task: "T1-01"
current_task_started: "2026-07-17T21:00:00+09:00"
consecutive_failures: 0       # 停止条件カウンタ
daily_jules_tasks_used: 3     # 予算カウンタ（日次リセット）
last_review:
  pr: 9
  verdict: approved            # approved / rejected / needs_split
  notes: "..."
pending_queue:                 # 設計済み・未投入のタスク
  - id: "..."
    title: "..."
```

**設計判断**: state を GitHub 上に置くことで、(a) 司令塔がどのマシンから起動しても状態を復元できる、(b) 人間がブラウザから現況を監視できる、(c) 履歴が git log に残る。

## 4. 司令塔の判断フロー（詳細）

各起動時、司令塔は以下を順に評価し、**最初に該当したアクションだけ**を実行して終了する（1起動1アクション原則。消費を抑え、判断を単純に保つ）。

| 優先 | 条件 | アクション | 消費 |
|---|---|---|---|
| 1 | state が paused/stopped | 即終了 | 極小 |
| 2 | 停止条件に該当（§6） | state を stopped に変更 + 通知 Issue 作成 | 小 |
| 3 | Jules セッション実行中 | 即終了（見守り） | 極小 |
| 4 | タイムアウト検知（タスク開始から⬜60分超） | セッション調査 → 再投入 or 失敗記録 | 中 |
| 5 | 未レビューのマージ済みPRあり | コードレビュー実施 → 合否を state に記録 | 大 |
| 6 | レビュー不合格が記録済み | 修正タスクを設計して投入 | 中 |
| 7 | pending_queue にタスクあり | 先頭タスクを tasks.yml に投入 → Jules 起動 | 小 |
| 8 | queue 空 & ロードマップに残項目 | ロードマップから次の2〜4タスクを設計 → queue へ | 大 |
| 9 | ロードマップ完遂 | state を paused に + 完了報告 Issue | 小 |

**レビュー観点（項目5の中身）**: 憲法v2に記載（別文書）。差分の妥当性 / テスト有無 / protected_paths 違反 / 不要ファイル（.egg-info 等、知見16）/ セキュリティ基本チェック。

## 5. 予算設計

| リソース | 上限 | 根拠 |
|---|---|---|
| Jules タスク | ⬜ 1日12タスク（案） | プラン上限100/日の12%。まず控えめに |
| Claude Code 消費 | レビュー・設計は1日⬜8回まで | Pro の5時間ローリング枠を温存 |
| ポーリング | 30分間隔 = 48回/日 | 大半は「即終了」で消費極小 |
| 深夜帯 | 制限なし（24時間稼働） | Jules 側は時間帯無関係 |

Pro 枠保護の実装: state.yml の `daily_heavy_actions` カウンタで重いアクション（項目5・8）を制限。超過時はアクションせず終了し、翌起動に持ち越す。

## 6. 安全弁・停止条件

以下のいずれかで `loop_status: stopped` に遷移し、通知 Issue を立てる:

1. **連続失敗**: 同一タスクが2回連続失敗（consecutive_failures >= 2）
2. **予算超過**: Jules 日次上限到達（この場合は stopped でなく「翌日まで待機」）
3. **CI 恒常的失敗**: main の CI が⬜3回連続で赤
4. **protected_paths 違反検知**: Jules の PR が保護領域に触れていた場合（レビューで却下 + 停止）
5. **人間による停止**: state.yml の loop_status を手動で paused に書き換え（緊急停止スイッチ）

**通知方法**: GitHub Issue 作成（タイトルに [LOOP-ALERT]）。⬜メール通知は GitHub の Watch 設定で代替可能か検証。

**破壊防止の多層構造**:
- 層1: protected_paths（Jules がパイプライン自体を触れない）
- 層2: branch protection + CI 必須（壊れたコードは main に入らない）
- 層3: 司令塔レビュー（CI を通る低品質コードを弾く）
- 層4: 停止条件（異常の連鎖を断つ）
- 層5: 人間の週次確認（方向性の逸脱を正す）

## 7. 認証・実行環境

| 項目 | 決定 | 備考 |
|---|---|---|
| 司令塔の実行場所 | **ローカルPC（タスクスケジューラ）** | Pro サブスク枠を使うため。GitHub Actions での実行は API 従量課金になるため不採用 |
| 起動スクリプト | commander.ps1（PowerShell） | PYTHONIOENCODING=utf-8 必須（知見5） |
| Claude Code 呼び出し | `claude -p "<司令塔プロンプト>" --allowedTools ...` | ⬜許可ツールの最小セットを PoC で確定 |
| gh CLI | **ローカルに要インストール**（現状未、知見12の環境変更） | 司令塔が PR/CI 状態を読むために必須 |
| PC スリープ問題 | ⬜要検討: 電源設定 or タスクスケジューラのスリープ解除オプション | 24時間稼働の物理的前提 |
