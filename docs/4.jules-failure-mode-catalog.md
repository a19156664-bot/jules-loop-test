# Jules 想定外動作カタログと対応策（草案 v0.1）
作成: 2026-07-17 / 建築家: Fable 5
用途: ループエンジニアリング設計の付属文書。憲法v2のレビュー手順・停止条件の根拠資料
凡例: 🟢実装済み / 🟡設計済み（未実装） / 🔴未対応（要検討）

---

## カテゴリA: 成果物の品質問題

### A-1. タスク要件の一部未実装
- **想定される動き**: プロンプトの要件7項目のうち5項目だけ実装してPRを作る
- **検知**: 🟡 司令塔レビュー（憲法v2 レビュー手順1: 要件充足チェック）
- **対応**: 🟡 rejected → 不足分を修正タスクとして再投入（判断表6）
- **予防**: 🟢 プロンプトに要件を番号付きリストで明記（既に実施中）

### A-2. 要件外の余計な実装（スコープ超過）
- **想定される動き**: 「ついでに」認証機能やログ機能を追加する
- **検知**: 🟡 司令塔レビュー（差分がpathsの範囲内か、要件と対応しているか）
- **対応**: 🟡 軽微なら approved + 記録、大きければ rejected
- **予防**: 🟢 プロンプトの「やらないこと」セクション（既に実施中、憲法v2で必須化）

### A-3. 不要ファイルの混入
- **想定される動き**: `.egg-info` / `__pycache__` / エディタ一時ファイルをコミット（**実績あり**、知見16）
- **検知**: 🟡 司令塔レビュー手順3
- **対応**: 🟢 Jules自身がコードレビューで自己修正した実績あり / 🟡 漏れたら除去タスク投入
- **予防**: 🔴 `.gitignore` に `*.egg-info` `__pycache__/` を追加（**4夜目前に実施推奨・5分作業**）

### A-4. CIは通るが低品質なコード
- **想定される動き**: テストが形骸化（assert True 等）、エラー処理なし、既存コードとの重複
- **検知**: 🟡 司令塔レビュー手順4・6
- **対応**: 🟡 rejected + 具体的指摘を notes に記録して修正タスク化
- **予防**: 🟡 3夜目のsmoke test導入でCI自体の網を強化（進行中）

### A-5. 既存機能の破壊（デグレード）
- **想定される動き**: GET /tasks を修正する際に /health を壊す
- **検知**: 🟢 CI（pytest が全エンドポイントを検査、3夜目T2以降）
- **対応**: 🟢 CI失敗 → auto-merge されない → 🟡 司令塔が判断表4/6で対処
- **予防**: 🟢 タスク粒度の制限（1〜3ファイル、+200行以内、憲法v2）

## カテゴリB: パイプライン領域への侵入

### B-1. protected_paths への変更
- **想定される動き**: `.nightly/` や `.github/workflows/` を「改善」しようとする
- **検知**: 🟢 nightly.py の gate（protected_paths チェック） / 🟡 司令塔レビュー手順2（二重化）
- **対応**: 🟡 即 rejected + **ループ停止**（憲法v2 停止条件3。最重要リスクのため）
- **予防**: 🟢 tasks.yml の protected_paths 定義 + プロンプトで対象ファイルを明示

### B-2. 憲法・state への干渉（司令塔導入後）
- **想定される動き**: COMMANDER.md / state.yml / ROADMAP.md を書き換える
- **検知・対応**: B-1 と同じ（protected_paths 配下に置く設計のため）
- **予防**: 🟡 4夜目以降、これらのファイルを protected_paths に追加登録

### B-3. ブランチ運用の逸脱
- **想定される動き**: main に直接push / 想定外のブランチ名でPR作成
- **検知**: 🟢 branch protection（integration/nightly-* に ci 必須）※mainへの直接pushはJulesの権限設計上起きにくいが
- **対応**: 🔴 main 自体には protection なし（朝レポート書き込みのため）。**Julesのbot権限でmainに直接pushできるかは未検証** → 🔴 要確認事項
- **予防**: 🔴 検証の結果次第で「mainへのpush制限 + 朝レポート用の例外」を検討（優先度中）

## カテゴリC: タスク実行の失敗・停滞

### C-1. タスク失敗（PR作成に至らない）
- **想定される動き**: 環境構築エラー、依存解決失敗、プロンプト誤解で空振り
- **検知**: 🟡 司令塔のタイムアウト検知（判断表4: 開始から60分⬜超過）
- **対応**: 🟡 1回だけ再投入 → 再失敗で consecutive_failures=2 → ループ停止 + [LOOP-ALERT]
- **予防**: 🟢 Jules は自動リトライ機能を持つ（公式仕様） + プロンプトの具体化

### C-2. タスクの無限作業（終わらない）
- **想定される動き**: テストが通らず修正を延々と繰り返す
- **検知・対応**: C-1 と同じ（タイムアウトで一律に扱う）
- **予防**: 🟢 15分粒度のタスク設計（実績4〜11分に対し60分⬜のタイムアウトは十分な余裕）

### C-3. 部分完了PR（タスクの半分だけ実装してPR作成）
- **想定される動き**: 難所で妥協し「TODO」コメント付きでPRを出す
- **検知**: 🟡 司令塔レビュー手順1
- **対応**: 🟡 needs_split 判定 → タスクを分割して再設計（憲法v2に定義済み）
- **予防**: 🟡 タスク粒度基準の遵守

### C-4. depends_on の無視（T1完了前にT2の前提を壊す）
- **想定される動き**: —
- **現状**: 🟢 構造的に発生しない。turn-switch がT1マージ後にT2を投入する直列設計のため
- **注意**: 🔴 将来の並列化（2列運用）時には再検討が必要

## カテゴリD: 想定外の出力・挙動

### D-1. スキーマ・規約の独自解釈
- **想定される動き**: tasks.yml を独自形式で書き換える、命名規則を無視する（**類似実績**: models.py の prompt/prompt_file 不整合）
- **検知**: 🟢 nightly.py validate / CI / 🟡 司令塔レビュー
- **対応**: 🟡 rejected + スキーマを明示した修正タスク
- **予防**: 🟢 AGENTS.md にスキーマ・規約を記載（Julesは AGENTS.md を自動で読む公式仕様）

### D-2. シークレット・危険コードの混入
- **想定される動き**: サンプルとしてAPIキー風文字列をハードコード、eval/exec の使用、外部への通信コード
- **検知**: 🟡 司令塔レビュー手順5（セキュリティ基本チェック）
- **対応**: 🟡 即 rejected。実キーの場合は人間に通知（キーのローテーション判断）
- **予防**: 🟢 リポジトリにシークレットを置かない運用（Jules公式も推奨） / 🔴 GitHub Secret scanning の有効化確認（public化により自動有効のはず、**5分で確認可能**）

### D-3. 大量ファイル生成・巨大diff
- **想定される動き**: node_modules 相当の依存ディレクトリや生成物を丸ごとコミット
- **検知**: 🟡 司令塔レビュー（+200行基準の大幅超過）
- **対応**: 🟡 rejected
- **予防**: 🟢 .gitignore 強化（A-3と同じ対応）

### D-4. PRを作らずセッション終了（成果消失）
- **想定される動き**: 作業完了したがPublish PRに失敗
- **検知**: 🟡 タイムアウト検知（PR無し=成果なし扱い）
- **対応**: 🟡 再投入（C-1と同じ経路）
- **予防**: 🟢 automationMode: AUTO_CREATE_PR 実装済み（初夜から安定動作）

## カテゴリE: 外部要因（Jules起因でないがJulesに影響）

### E-1. Jules サービス障害・メンテナンス
- **検知**: 🟡 タイムアウト検知に吸収される
- **対応**: 🟡 consecutive_failures 経由でループ停止 → 人間が状況確認して再開
- **予防**: 不可（外部サービス）。🟢 停止→安全待機の設計で被害ゼロ化

### E-2. プラン枠の枯渇（日次100タスク / 並列10）
- **検知**: 🟡 state.daily_jules_tasks_used カウンタ（憲法v2 予算管理）
- **対応**: 🟡 「翌日まで待機」（stopped にしない設計済み）
- **予防**: 🟡 日次上限12⬜の自主制限（枠の12%運用）

### E-3. GitHub API レート制限・Actions障害
- **検知**: 🔴 司令塔の gh コマンド失敗として顕在化 → 現状ハンドリング未定義
- **対応**: 🔴 **要追加設計**: gh 失敗時は「stopped ではなく次回起動に持ち越し」を憲法v2に追記推奨
- **予防**: 30分ポーリングなら通常レート制限には達しない

## 対応の優先順位まとめ

### 今すぐ（4夜目前・各5分）🔴→🟢化
1. `.gitignore` に `*.egg-info` / `__pycache__/` を追加（A-3, D-3）
2. GitHub Secret scanning の有効化確認（D-2）

### 憲法v2への追記(次回改訂時)
3. gh コマンド失敗時の「持ち越し」ルール(E-3)
4. protected_paths に COMMANDER.md / state.yml / ROADMAP.md を追加する手順(B-2)

### 検証事項(PoC時)
5. Jules の bot 権限で main に直接 push できるか(B-3)
6. タイムアウト閾値の実測確定(C-1, C-2)

### 設計思想の確認
本カタログの大半は「検知=司令塔レビュー、対応=rejected+修正タスク、最悪時=ループ停止」に収斂する。
つまり**個別の想定外に個別対処するのではなく、「レビュー網 + 停止条件」の2つを堅牢にすることが、未知の想定外への最良の備え**である。
既知15ケース中、ループ停止まで到達しうるのは B-1/B-2(侵入)と C-1/C-2(連続失敗)のみで、他はすべて1回転以内に自動回復する設計になっている。
