# Fable 5 × Google Jules リレー運用ガイド
### 〜建築家(設計・検収)と非同期コーディングエージェント(実装)の分業プロセス〜

> 三部作フレームワークの初の実地適用。第2部「行動の型」のリスク三分類と、第3部「統率の型」の検収・差し戻し原則を、Julesの実仕様にマッピングしたものです。

---

## 1. 役割分担への見解 — 「効果的。ただし4つの前提条件付き」

### 結論
「Fable 5が基本設計 → JulesがPR提出 → Fable 5がレビュー」のリレーは効果的です。Julesの成果物が**必ずPRという検証可能な形式で出てくる**点は、第3部・原則1(境界は検証可能な成果物で切る)を製品仕様として満たしており、相性は良好です。ただし以下の4点を押さえないと「丸投げ」は事故ります。

### 前提条件① リレーは2段階ではなく「3段階」にする
Julesは実装前に計画を提示し、承認を待つ仕組みを持ちます(放置すると自動承認される点に注意)。この**計画承認を建築家の中間検収ゲートとして必ず使ってください**。

```
[設計] Fable 5がIssue(指示書)を作成
   ↓
[計画検収] ★Julesが提示したPlanをFable 5に貼ってレビュー(数分)
   ↓ 承認 or 計画修正の指示
[実装] Julesが数時間の自律実装(ここは放置)
   ↓
[成果検収] PRのDiffをFable 5がレビュー → 承認 or 診断付き差し戻し
```

計画段階の誤りは数分で直せますが、実装後の方向性の誤りは数時間の作業と1タスク枠を無駄にします。「放置できる」のは計画承認の後からです。

### 前提条件② タスクは小さく切る(建築家の最重要責務)
Julesは明確にスコープされた小タスクに強く、大規模リファクタや多数のファイルが相互依存する変更は苦手です。またタスク間の連携ができないため、依存関係のある変更は**建築家(Fable 5)が直列に分解し、順番に投入する**必要があります。分割の目安:

- 1タスク = 1PR = 1つの検証可能な完了条件群(第3部・原則1)
- 「このPRだけをレビューして合否が判断できるか?」で切る
- 依存する2つの変更を1タスクに詰めない。前のPRがマージされてから次を投入する

### 前提条件③ AIレビューを唯一の防衛線にしない
レビュアーとしての私は**静的レビュー(Diffを読む)しかできず、コードを実行できません**。防衛線は三層で構成してください:

1. **CI(機械)**: テスト・Lint・ビルドの機械的検証。JulesのPRには既存CIが走るので、これを必須化する
2. **Fable 5(AI)**: 設計意図との整合、範囲逸脱、ロジックの静的検証(後述のレビュープロンプト)
3. **人間**: マージボタンは人間が押す。第2部の分類でいえばマージは [WRITE-I](不可逆操作)であり、AIに委譲しない

### 前提条件④ レビュアーの文脈不足を人間が補う
私はリポジトリ全体を直接見られません。Diffだけ渡されると「変更箇所は正しいが、他のファイルとの整合が壊れている」類の問題を見逃します。レビュー依頼時に渡すべき文脈はセクション3に定義しました。

### 想定される失敗モードと対策(プレモーテム)

| 失敗モード | 原因 | 対策 |
|-----------|------|------|
| 動くが設計意図と違うPR | Issueの完了条件が曖昧 | セクション2の雛形(検証可能な完了条件を必須化) |
| 依頼範囲外のファイルまで変更 | スコープ境界の未定義 | Issueに「触ってよい/いけないファイル」を明記 |
| テストが通っているように見えて実質空 | テストを実装に合わせて改変 | レビュー観点に「テストの実質性」を含める(セクション3) |
| 計画が自動承認されて暴走 | 承認待ちの放置 | 計画通知を必ず確認する運用 or 計画不要な軽微タスクに限定 |
| 差し戻しの無限ループ | 指示側の欠陥をJulesの能力問題と誤認 | 差し戻し2回で人間が指示書側を見直す(第3部・原則7) |

## 2. 指示の最適化 — 「二層の指示書」で暴走を防ぐ

Julesへの指示は**恒久ルール(AGENTS.md)**と**個別指示(Issue)**の二層に分けます。JulesはリポジトリルートのAGENTS.mdを自動で読み込むため、毎回のIssueに書くべきでない共通規範はそちらに置きます。これは第3部・原則4「全員が同じ憲法を持つ」のJules版実装です。

### 層1: AGENTS.md(リポジトリに常設する「憲法」)

```markdown
# AGENTS.md

## プロジェクト概要
{{技術スタック、アーキテクチャの1段落説明}}

## コーディング規約
- {{スタイルガイド、命名規則、使用禁止ライブラリ}}
- 新規依存パッケージの追加は、Issueで明示的に許可された場合のみ

## テスト
- テスト実行コマンド: {{例: npm test}}
- 変更したロジックには必ず対応するテストを追加すること
- 既存テストを削除・スキップ・期待値変更してはならない。
  既存テストが失敗する場合は、修正せず失敗内容を報告すること

## 禁止事項(フェイルセーフ)
- 触ってはいけないパス: {{例: /migrations, /config/production, .github/workflows}}
- 認証情報・APIキー・シークレットをコードやログに書き込まない
- Issueに記載のないファイルの大規模な書き換え(リフォーマット含む)を行わない

## 報告義務
- 指示に曖昧さや矛盾を発見した場合、推測で埋めずにPR説明文に「未確定事項」として明記すること
```

> ポイントは「既存テストの改変禁止」です。エージェントが「テストを直してグリーンにする」のは最も検出しにくい失敗モードであり、憲法レベルで封じます。

### 層2: Issue雛形(タスクごとの指示書)

第1部の思考の型(目的とWhatの分離、完了条件)と第2部のフェイルセーフをIssueに埋め込んだ雛形です。

```markdown
## 目的(Why)
{{この変更がなぜ必要か。1〜3文。Julesが実装判断に迷った際の判断基準になる}}

## 変更内容(What)
{{何をどう変えるか。具体的に}}

## 完了条件(Definition of Done)— すべて検証可能な形で
- [ ] {{例: POST /api/users に不正なemailを送ると400とエラーコードE-102を返す}}
- [ ] {{例: 上記の正常系・異常系のテストが追加され、npm test が全件パスする}}
- [ ] {{例: 既存のAPIレスポンス形式に変更がない(後方互換)}}

## スコープ
- 変更してよいファイル/ディレクトリ: {{明示列挙。「主に」ではなく確定で}}
- 変更してはいけないもの: {{列挙。AGENTS.mdの禁止事項に加える個別指定}}
- 今回やらないこと: {{隣接する改善の誘惑を先回りして封じる。例: リファクタは別Issue}}

## 技術的制約・ヒント
- {{使用すべき既存関数・パターン、参照すべき類似実装のファイルパス}}
- {{パフォーマンス・互換性などの非機能要件}}

## 詰まった場合の指示(フェイルセーフ)
- 完了条件を満たせない障害に遭遇したら、回避策を発明せず、
  できた部分までで作業を止め、障害の内容をPR説明文に記載して提出すること
- 本Issueの指示とコードの実態が矛盾していた場合も同様に、報告を優先すること

## PR説明文に含めるべき項目
1. 完了条件との対応表(各条件を満たした方法)
2. 変更ファイル一覧と各変更の理由
3. テスト実行結果
4. 未確定事項・妥協した点(なければ「なし」と明記)
```

### 雛形の設計意図(なぜこう書くか)

1. **完了条件が指示書の心臓部。** 「バリデーションを改善する」ではなく「不正emailで400を返す」と書く。曖昧な完了条件は、暴走ではなく「もっともらしい別物」を生みます。これが的外れPRの最大原因です。
2. **「やらないこと」の明記が範囲逸脱を防ぐ最安の保険。** エージェントは隣接コードの「ついで改善」をしがちで、それがレビューコストとコンフリクトを膨らませます。
3. **「詰まったら止まって報告」は第2部・原則10(撤退条件)の移植。** 非同期エージェントは途中で質問できない分、「創意工夫で突破」より「正直な部分提出」を明示的に上位に置く必要があります。
4. **PR説明文の様式指定は、レビュアー(Fable 5)への引き継ぎスキーマ。** 第3部・原則3(構造化通信)の実装であり、これがあるとレビュー精度が大きく上がります。

## 3. レビューの型 — Fable 5への依頼プロンプト

### 渡すべき文脈(4点セット)
Diffだけでは範囲逸脱や整合性破壊を検出できません。以下を毎回セットで渡してください:

1. **元のIssue全文**(完了条件との照合がレビューの主軸のため必須)
2. **PRのDiff全文**とJulesが書いたPR説明文
3. **CIの実行結果**(パス/失敗、テスト件数)
4. **AGENTS.mdの内容**(初回のみ。以降は「変更なし」でよい)

必要に応じて、Diffが参照している周辺ファイル(呼び出し元など)を追加で渡すと精度が上がります。

### コピペ用レビュー依頼プロンプト

```markdown
あなたはこのプロジェクトの設計者(建築家)として、配下のコーディングエージェント
「Jules」が提出したPRを検収してください。あなた自身がこのIssueの設計者である
という立場で、実装が設計意図に合致しているかを裁定します。

## 入力
### 元のIssue(指示書)
{{Issue全文を貼付}}

### AGENTS.md(プロジェクト憲法)
{{内容を貼付 / 前回から変更なし}}

### PR説明文(Jules記載)
{{貼付}}

### CI結果
{{パス/失敗、テスト件数、ログ抜粋}}

### Diff
{{diff全文を貼付}}

## レビュー観点(この順で全項目を検査すること)
1. 要件充足: Issueの完了条件を1項目ずつ照合し、各項目に 合格/不合格/検証不能 を付ける
2. 範囲逸脱: Issueで許可されていないファイル・機能への変更がないか。
   「ついで改善」も逸脱として指摘する
3. テストの実質性: 追加テストは完了条件を本当に検証しているか。
   既存テストの削除・スキップ・期待値の改変がないか(最重要監査項目)
4. 整合性・回帰リスク: 変更が呼び出し元・依存先との契約(引数、返り値、
   例外、レスポンス形式)を壊していないか。Diff外への影響が疑われる箇所を挙げる
5. 品質・セキュリティ: 明白なバグ、エラー処理の欠落、入力検証漏れ、
   認証情報のハードコード、インジェクション脆弱性
6. 報告の誠実性: PR説明文の主張とDiffの実態が一致しているか。
   「未確定事項なし」と書きながら妥協の痕跡がないか

## 制約
- あなたはコードを実行できない。静的に検証できない項目は「合格」とせず
  [検証不能: 人間による動作確認が必要] と明示すること
- Diffに含まれない部分の動作は推測で保証しないこと
- 確信度ラベル [事実/推論/仮説] を重要な指摘に付けること

## 出力形式
1. 判定: APPROVE / REQUEST_CHANGES / HUMAN_REVIEW_REQUIRED のいずれか
   (完了条件に1つでも不合格があれば APPROVE 不可。
    設計判断や仕様の曖昧さに起因する問題は HUMAN_REVIEW_REQUIRED)
2. 完了条件チェックリスト(1項目ずつ合否と根拠)
3. 指摘事項(重大度: Blocker / Major / Minor に分類し、該当行を引用)
4. REQUEST_CHANGES の場合: Julesにそのまま送れる修正指示文
   (未達の完了条件 + 具体的な不備 + 修正の方向性、の診断付き形式)
5. 人間が最終確認すべき事項(動作確認手順の提案を含む)
```

### 運用ルール(第3部の原則の適用)

- **差し戻しは診断付きで、最大2回。** REQUEST_CHANGES時の「修正指示文」はJulesのタスクへのフィードバックとしてそのまま送れる形式で出させます。2回で合格しなければ、問題はIssue(指示書)側にあると推定し、人間+建築家でIssueを書き直して新タスクとして再投入します(第3部・原則7)。
- **マージは人間。** 判定がAPPROVEでも、それは「静的レビュー合格」の意味です。CI合格+AI承認+人間の最終確認(と、必要なら動作確認)が揃ってマージ、が標準線です。
- **レビュー結果はログに残す。** 判定・指摘・差し戻し回数は第3部の「検収・裁定ログ」そのものです。差し戻しが特定種類のタスクに偏るなら、それはIssue雛形かAGENTS.mdを進化させる材料になります(データフライホイールの起点)。
